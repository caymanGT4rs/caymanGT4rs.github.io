Category 1: Software Design & Engineering Enhancements

The artifact I have chosen for the Software Design and Engineering part of the final project is an Android application I developed for the CS-360 course. The application is designed for users to be able to add items to a list that is associated with their login credentials and to be able to add, delete, or modify items to the list. I created this application last fall and was extremely happy with the proof-of-concept design and functionality of the application. I selected this artifact for the ePortfolio because of its complexity as well as the number of enhancements I can make to it to show the skills I have learned while being a student at SNHU. Since the application was a functional proof-of-concept app, the code as well as the UI/UX was very rough. However, this allows me to refine the code to be a much smoother experience not only for the user through creating an easier navigable UI and visually pleasing one, but also for any future developers that work on the code through creating concise code blocks and great readability without unnecessary code. This is why I have also selected this artifact for the other components that will be mentioned in the ePortfolio. For the enhancements on this area, I implemented a separate page for registration of a user. Previously, the login and registration used the same text box. This can be confusing for some new users since this is not what many of the other apps use. This separate registration page also adds a security measure by requiring a matching password to ensure that the user has entered the password they want correctly. Also, this new registration page allows for me to separate the login vs registration sequence and reduce unnecessary code. I have also added password visibility toggles for the login as well as the registration page that toggles both the initial password entry, and the password re-entry. This feature allows for the user to have a better experience when entering their password if they feel that they have messed up. I have also changed the “SMS Permissions” button into a “Settings page where the new SMS permissions are now homed. I have met the outcomes that I originally planned for in module one and have the additional outcome of “Developing a security mindset that anticipates adversarial exploits in software architecture and designs to expose potential vulnerabilities, mitigate design flaws, and ensure privacy and enhanced security of data and resources” with the features regarding the passwords that I implemented. I plan to further achieve this outcome with other enhancements in the future. While making the enhancements to the artifact in the Software Design and Engineering category, I learned how to more efficiently set up pages inside of the app to allow for more simplistic pages where users will have the most traffic. Previously, I had the options of registration and settings on the main login page and main display page. This made the previous UI look confusing and would give a bad experience to the user. While implementing the separate pages and adding features such as immediate password validation and visibility toggle, I was able to hide these features to where most users will not need to see it every time they go to use the app. Challenges I faced were figuring out the best way to have the features be implemented in the shortest amount of code. Initially, I was able to get my idea of the features I wanted to implement working great but took up a lot of code which for future development of the app may be inefficient. Since I want to future proof the code as much as possible, I started to remove code so that I could accomplish the same task using shorter code. I was able to have the password re-entry in the same code block as the registration validation while still maintaining good readability and encapsulation which allowed for the relevant code to be all in one spot and not feel crowded. The settings page I have implemented now has room for future buttons that can be stored in that page as well as the log-out button which takes the user to the main page. The current issue with the app now is that the items for each user are not being saved when navigating through the pages. For example, going from DataDisplayActivity to the SettingsActivity page and then back clears the list of items from the inventory list. This should be an easy enough fix that I will work on in the Data Retrieval part of the app.

Category 2: Algorithms & Data Structures Enhancements

The artifact is the same artifact used for the previous enhancement. The artifact I have chosen for the Software Design and Engineering part of the final project is an Android application I developed for the CS-360 course. The application is designed for users to be able to add items to a list that is associated with their login credentials and to be able to add, delete, or modify items to the list. I created this application last fall and was extremely happy with the proof-of-concept design and functionality of the application. For this reason, I have decided to include the application as part of the ePortfolio for this class. The skills I will demonstrate with this enhancement is how I can change the data structure of the current implementation and allow for better functionality without the handoff of worse performance. I plan on improving the data structure and adding a sorting function to allow for the items displayed in the item inventory list to be sorted by item ID and by item Quantity. Before enhancement, the app used the “DataItem.Java” class to put the created item into a Parcel with the fields: Name, Quantity, ID, and put the item parcel into an Array for the item list to display. My first enhancement is not major but will improve the performance of this item adding method. I initially had the option for the user to add their item with a preferred ID. After talking to friends that said they would use an app like this, they said that the feature would be one that they would not use. With this input, I removed the feature which is one less thing that needs to be processed. The item is still assigned an ID, however the item ID is now assigned in sequential order, incrementing from the last added item and not by filling in the lowest added number available. For this I moved the UniqueIdGeneration method to the DataDisplayActivity to prevent math moving back and forth between classes. After the maxId is found from the arrayList, it is sent to the AddDataActivity to be used as the nextId. With this, I was able to remove unnecessary code from both the AddDataActivity and the DataDisplayActivity. During the enhancement process, I learned more efficient ways to have array lists reordered in Java by using specific identifiers within the list. The problems I encountered involved the sorting method buttons and keeping the view on the Data Display. For this reason, I have decided to include the application as part of the ePortfolio for this class. The skills I will demonstrate with this enhancement is how I can convert the storage of data from arrays to be held into a SQL database. Currently, I have a SQL database setup for the login credentials of the users, with a table storing usernames in one column and passwords in the other column. I am wanting to convert the current array storage of the items to database so that I can store the inventory after the app closes as well as eventually link inventories to user ID’s so that each user will be able to have their own stored inventory. I will first add a table to the database labeled “inventory” with columns labeled ID; Name; Quantity, that will hold the inventory items with these data fields. Next, I will add the methods for accessing and modifying the table. I’ll need to load data from the table on startup of the application to the main screen of the inventory app. The current add button will need to be linked to a different add method. The same will need to be done for the delete method. Testing the enhancements worked well. I incremented the SQL database version integer in the DatabaseHelper class which restarted the onCreate method adding the new table and erasing the old information. I had forgotten that I needed to restart the database to create the table for the inventory items which stumped me for a while, but after reading through my code and seeing the onCreate being under a version number, I tested changing the number to version 2 and everything worked.
